# Ubiquitous Language

## Boss Content

| Term                          | Definition                                                                                                                                                                                                                                                                                                                                                                                                                    | Aliases to avoid                                |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| **Boss**                      | A raid encounter that drops a sellable crystal when defeated                                                                                                                                                                                                                                                                                                                                                                  | Raid, dungeon boss                              |
| **Boss Family**               | A group of boss difficulties that share a weekly entry slot                                                                                                                                                                                                                                                                                                                                                                   | Boss group, boss tier                           |
| **Boss Difficulty**           | A specific tier within a **Boss Family** (Easy, Normal, Hard, Chaos, Extreme)                                                                                                                                                                                                                                                                                                                                                 | Boss level, boss rank                           |
| **Crystal**                   | A tradeable item dropped by a defeated **Boss**, sold to a vendor for mesos                                                                                                                                                                                                                                                                                                                                                   | Boss drop, crystal drop                         |
| **Crystal Value**             | The meso amount received when selling a **Crystal** from a specific **Boss Difficulty**; stored per-**World Group** as `{ Heroic, Interactive }`, resolved at income time against the **Mule's** **World Group**                                                                                                                                                                                                              | Sell price, crystal price, meso value           |
| **Heroic Crystal Value**      | The **Heroic**-side component of a **Boss Difficulty's** **Crystal Value** — the meso amount a **Mule** on a Heroic world receives per sold **Crystal**                                                                                                                                                                                                                                                                       | Heroic price, heroic sell value                 |
| **Interactive Crystal Value** | The **Interactive**-side component of a **Boss Difficulty's** **Crystal Value** — the meso amount a **Mule** on an Interactive world receives per sold **Crystal**; typically ~20% of the **Heroic Crystal Value**, but not guaranteed per boss (see **Crystal Value Ratio**)                                                                                                                                                 | Interactive price, interactive sell value       |
| **Crystal Value Ratio**       | The informal `Interactive ÷ Heroic` relationship across the **Boss** matrix. ~0.2 (Interactive = 1/5 Heroic) for 75 of 76 `(boss, tier)` pairs. **Extreme Kaling** is the only current outlier (Interactive `1 205 250 000` vs a 5× implied `1 205 200 000`, a 250 000-meso gap) — accepted as a real per-boss deviation, not a typo, so the ratio is deliberately **not** asserted in tests. Future bosses may also break it | Interactive multiplier, 0.2 multiplier, 5x rule |
| **World Pricing**             | The two-value-per-tier **Crystal Value** model that makes a **Mule's** income reflect its **World Group** instead of a single unified price. The matrix stores both **Heroic Crystal Value** and **Interactive Crystal Value** independently, and income resolves one or the other via `resolveWorldGroup(mule.worldId)`                                                                                                      | Per-world pricing, world-aware pricing          |

## World & Server Model

| **World Weekly Crystal Cap** | The constant `180` — MapleStory's per-world per-week boss-crystal sale ceiling. | **Resolve World Group**      | The `resolveWorldGroup(worldId)` helper in `src/data/worlds.ts` that maps any `worldId` value — valid **World Id**, stale string, `null`, or `undefined` — to a **World Group**; defers to `findWorld` and falls back to **Fallback World Group** on miss. Single source of truth for the lookup (used by `Income.of`, `useBossMatrixView`, and `MuleBossSlate.from`)                                            | getWorldGroup, worldGroupOf                                  |

| Term                       | Definition                                                                                                                                                                                                                              | Aliases to avoid                   |

Dropped slots contribute zero meso and zero count to every cap-aware readout (**Total Weekly Income**, **Crystal Sale Tally**, **PieChart Card**, **KPI Card**'s WEEKLY/DAILY tiles)                                                                                                                            | Cap drop, top-180 cut, slot trim          |

| **Soft Cap**          | A cap that allows over-selection at the input but enforces the limit downstream. **World Weekly Crystal Cap** is the canonical example — the user may select more than 180 **Crystal Slots'** worth of bosses, and the **World Cap Cut** silently trims contribution to 180. | **Hard Cap**          | A cap that blocks over-selection at the input. **Weekly Crystal Cap** (`14` weekly **Slate Keys** per **Mule**) and **Monthly Crystal Cap** (`1`) are the canonical examples — `canToggle` rejects a 15th weekly, and the **Monthly Radio Mutex** rejects a second monthly. ## Relationships

- A **Boss Family** contains one or more **Boss Difficulties**
- A **Mule** selects at most one **Boss Difficulty** per **Boss Family** per **Weekly Reset** (mutual exclusivity)
- A **Boss Difficulty** has exactly one **Crystal Value** — but the **Crystal Value** itself carries one number per **World Group** (**Heroic Crystal Value** and **Interactive Crystal Value**), resolved at income time against the **Mule's** **World Group** via **Resolve World Group**
- **Heroic Crystal Value** ≥ **Interactive Crystal Value** for every **Boss Difficulty** — a structural invariant enforced by `bosses.test.ts` (gameplay truth: Interactive worlds pay less than Heroic per **Crystal**)
- The **Mule Boss Slate** is bound to one **World Group** at construction (`MuleBossSlate.from(keys, worldGroup)`); every price-reading method (`view`, `totalCrystalValue`, selection-conflict tie-breaking) resolves `BossDifficulty.crystalValue[worldGroup]` through that binding. Each **World Group** has its own reference-stable empty slate via `MuleBossSlate.emptyFor(worldGroup)`, so `slate === MuleBossSlate.EMPTY` (Heroic) or `===` the Interactive empty singleton remains a cheap React-memoization check
- `Income.of` resolves each **Income Source's** **World Group** individually — a roster mixing Heroic and Interactive **Mules** prices each one against its own **Crystal Value** component
- A **Mule** with no **World Id** (legacy pre-**World Select** persistence) is priced under **Fallback World Group** (Heroic), preserving its pre-**World Pricing** income exactly
- A **Mule** has one **Potential Income** = sum of selected **Crystal Values**
- A **Mule** is an **Active Mule** iff its **Active Flag** is `true` (previously: iff **Potential Income** > 0; see "Flagged ambiguities")
- **Total Weekly Income** = sum of all **Active Mules'** **Potential Incomes** — an **Inactive Mule** contributes zero regardless of its boss selection
- An **Inactive Mule's** **Character Card** is rendered in the **Dim State** (0.55 opacity); its income line still shows the **Mule's** **Potential Income** but in the muted color
- **Computed Value** = `crystalValue` for **Daily Cadence** (**Party Size** ignored, matching **Boss Matrix**'s cell rule); `crystalValue / partySize` for **Weekly Cadence** and **Monthly Cadence**. **Party Size** defaults to `1` when absent. **Total Crystal Value** is a plain sum of these — the **Weekly Crystal Cap** invariant on `MuleBossSlate.from` already caps weeklies at 14 per slate, so no income-time cut is needed
- A **Character Card** represents exactly one **Mule** in the **Roster**
- Clicking a **Character Card** or a **PieChart Card** slice opens the **Drawer** for that **Mule**
- The **Drawer** contains exactly one **Boss Matrix**, sitting under one **Matrix Toolbar** and one **Boss Search**
- The **Matrix Toolbar** renders above the **Boss Search**, which is **Fused** to the top of the **Boss Matrix**
- A **Canonical Preset** is **Active Preset** iff the **Mule's** weekly **Slate Keys** satisfy **Same-Cadence Equality** against its **Preset Entries** — one key per entry with tier in **Accepted Tiers**, and zero weekly keys on families outside the preset
- The **Custom Preset** is **Active Preset** iff the **Mule** has ≥1 weekly **Slate Key** and no **Canonical Preset** is **Active Preset**; if the weekly selection is empty, no **Preset Pill** is the **Active Pill** regardless of daily selections
- At most one **Boss Preset** is **Active Preset** at any moment — **Canonical Presets** are mutually exclusive under **Same-Cadence Equality**, and **Custom Preset** is mutually exclusive with every **Canonical Preset** by construction
- The **Active Pill** is derived from the **Boss Slate** on every render — no "last clicked" state is persisted. Clicking **Custom Preset** and then closing+reopening the **Drawer** restores whatever **Canonical Preset** still matches the underlying **Boss Slate**
- **CRA** ∩ **CTENE** share Vellum, Crimson Queen, Papulatus, and Magnus; **CRA** ⊂ **LOMIEN** family-wise. Under **Same-Cadence Equality** these intersections never cause two **Canonical Presets** to be **Active Preset** simultaneously — a selection matching LOMIEN's 14 entries can't also equal CRA's 11
- LOMIEN's **Multi-Tier Entries** (Lotus, Damien) mean the **Mule** can swap between Normal and Hard manually on those families without falling out of **Active Preset** status
- The **Cadence Filter** and **Boss Search** query reset to defaults each time the **Drawer** opens for a **Mule**. The **Active Pill** does not "reset" — it is derived each render from the current **Boss Slate**
- **Matrix Reset** clears `Mule.selectedBosses` only; **Cadence Filter** and **Boss Search** persist across the reset. The clicked **Canonical Preset** does not change the **Cadence Filter** — a user viewing the Daily filter when they click CRA sees no grid change for weekly-only families, only **Active Pill** feedback
- **Weekly Count** is bounded by **Weekly Crystal Cap** by construction — `MuleBossSlate.from` trims any over-cap weekly **Slate Keys** at slate construction time (lowest-`crystalValue` first, ties broken by insertion order). The **Crystal Tally's** Weekly cell never reads above `14/14` for any slate the live system can produce
- The **Drag Boundary** is visible only while dragging within the **Roster**
- A **Character Card** cannot be dragged outside the **Roster** (confined)
- Exactly one of the default **Roster Header** or the **Bulk Action Bar** is visible at a time — they occupy the same slot
- Entering **Bulk Delete Mode** disables drag-to-reorder (dnd-kit sensors suspended) and hides the **Add Card**
- In **Bulk Delete Mode** a **Character Card's** click toggles it as a **Deletion-Marked Mule** instead of opening the **Drawer**; the hover trash popover and the level badge on the **Character Card** are hidden so the **Selection Indicator** has room
- **Bulk Confirm** calls `deleteMules(ids)` on the set of **Deletion-Marked Mules** and then exits **Bulk Delete Mode**; there is no confirmation modal — selection itself is the confirmation
- **Bulk Cancel** exits **Bulk Delete Mode** without removing any **Mule**
- The trigger of **Bulk Confirm** is rendered in two mutually exclusive locations under the **Touch Device Detector**: on non-touch it is the in-bar `Delete` button (disabled while `selectedCount === 0`); on touch the in-bar button is omitted and the **Bulk Delete Pill** is rendered instead (only when `selectedCount > 0` — there is no disabled state, absence is the affordance)
- The **Bulk Delete Pill** is portaled to `document.body` via `createPortal` because the page-level `<section>` wrapping the **Roster** uses `animate-in slide-in-from-bottom-4`, whose residual `transform` would otherwise create a containing block that re-anchors the pill's `position: fixed` away from the viewport. Without the portal the pill scrolls with the page instead of staying viewport-anchored
- The **Bulk Delete Pill's** background is `var(--destructive)` with white text — deliberately matching the **Selection Indicator's** filled-checkbox background, so the floating affordance reads as the same destructive surface the user has been tapping on cards. The colour is opaque (no transparency or `color-mix` blend) so the pill never reads "see-through" against scrolled content
- The **Touch Bulk Layout Swap** flips which of the two header artefacts hides at `max-[524.99px]`: on non-touch the instructional text "Select or drag to delete" hides and the **Bulk Selection Pill** chip stays; on touch the chip hides and the text stays. The chip is redundant on touch because the count is already on the **Bulk Delete Pill**, and the text is the more useful anchor when the bar is decoupled from its primary CTA
- A `body:has([data-bulk-delete-pill])` rule at viewports ≤600px raises sonner's `[data-sonner-toaster][data-y-position='bottom']` `bottom` to `calc(env(safe-area-inset-bottom) + 4rem)` so toast notifications never sit beneath the pill. The `:has()` selector outweighs sonner's stamped specificity without `!important`. The same rule fires for `body:has([data-mule-detail-drawer])` (the **Drawer Close Pill's** anchor) so the toast offset works for either floating pill
- A `body:has([data-bulk-delete-pill])` rule also adds `padding-bottom: calc(env(safe-area-inset-bottom) + 6rem)` so at maximum scroll the last **Character Card** has clearance from the pill. Inherits the `selectedCount > 0` gate by design — runway only appears once at least one card is **Deletion-Marked**
- The **Drawer Close Pill** renders only `<sm` (`sm:hidden`) — desktop dismisses the **Drawer** via the sheet backdrop or Esc, where a sticky pill would just steal real estate. The pill triggers **Drawer Close** through the same `onClose` callback the pre-pill in-flow Close button used; no new state path
- The **Single Delete** path and **Bulk Delete** path both ultimately mutate the same `mules` state; **Single Delete** uses `deleteMule(id)`, **Bulk Delete** uses `deleteMules(ids)`
- **Drag-to-Select** is only available inside **Bulk Delete Mode**; outside **Bulk Delete Mode** a drag on the **Roster** is either a reorder gesture (dnd-kit) or a no-op on text
- A **Drag-to-Select** gesture engages on the first `pointermove` after `pointerdown`; a `pointerdown` → `pointerup` with zero movement falls back to the single-toggle click path on the **Start Card**
- During an active **Drag-to-Select**, exactly the **Mules** whose array index lies within the **Paint Range** are set to the **Brush** state; all other **Mules** hold their **Original Snapshot** value, including any **Mule** the gesture previously passed over but has since backed out of
- The **Start Card** is always inside the **Paint Range** and therefore always at the **Brush** state for the entire gesture
- **Gesture Cancellation** reverts every **Mule** touched by the gesture to its **Original Snapshot**; `pointerup` after engagement commits the current **Paint Range** as the final set of **Deletion-Marked Mules**
- **Drag-to-Select** requires `touch-action: none` and `user-select: none` on every **Character Card** within the **Drag Boundary** while **Bulk Delete Mode** is active, to suppress text highlighting and native touch scroll during the gesture
- The **PieChart Card** includes only **Active Mules** — an **Inactive Mule** renders no donut slice
- An **Inactive Mule's** **Character Card** remains fully interactive: hover-lift, click-to-open **Drawer**, and drag-to-reorder all work identically to an **Active Mule**; only the **Dim State** differs
- The **Active Toggle** sits on its own row directly below the weekly-mesos pill in the **Identity Section**; clicking it mutates the **Mule's** **Active Flag**. Both surfaces share `px-3 py-1.5` and a 1px border, so the Active Toggle gets `minHeight: 38` to match the weekly-mesos pill's natural ~38px height (which is dominated by the meso value's `text-base` line-height) — without it the toggle reads visibly shorter at 30px
- The **Drawer's** top-right ghost trash button bumps from `size-7` (28px) to `size-9` (36px) at the same `@max-[599.99px]/drawer` breakpoint where the **Crystal Tally** flips horizontal — once the tally row spans the full drawer width the smaller icon-sm button reads tiny against it
- Exactly one **Theme** and one **Density** are active at any time; both persist to localStorage
- The **Chart Palette** is a property of the active **Theme** — slice colors recolor when the **Theme** changes
- The **Density Toggle** mutates the active **Density**, which drives CSS variables that resize the **Roster** columns, **Character Card** padding, name size, and minimum card height
- The **Reset Countdown** targets the next **Reset Anchor** after `Date.now()`; the countdown value updates once per second from a single `setInterval`
- The **Reset Countdown** uses the **Live Countdown Format** at the `sm` breakpoint and above, and the **Smart Countdown Format** below it
- At **Reset Anchor** crossover the **Reset Countdown** silently rolls over to the next week — no flash, no announcement
- A **World** belongs to exactly one **World Group** (**Heroic** or **Interactive**); **World Ids** are globally unique across both groups
- A **World** belongs to exactly one **Region** (**NA Region** or **EU Region**); **Region** is orthogonal to **World Group**, so each `(World Group, Region)` combination is independent — **Solis** is `(Heroic, EU)`, **Luna** is `(Interactive, EU)`, the other four **Worlds** are `(_, NA)`
- The **Selected World** persists to localStorage under the `'world'` key; an unrecognised stored **World Id** falls back to `null` rather than crashing
- **World Provider** sits outside **Income Provider** in the component tree (`ThemeProvider > DensityProvider > WorldProvider > IncomeProvider > Dashboard`) so income consumers can read `useWorld()` without restructuring
- A **Mule** belongs to exactly one **World**, identified by **Mule World Id**; the binding is set at creation and immutable — the only way to move a **Mule** between **Worlds** is to delete and recreate it
- The **World Lens** predicate is `mule.worldId === selectedWorld.id`; a **Mule** with no **Mule World Id** (legacy pre-lens persistence) never matches any lens and is therefore invisible across every **Selected World** until deleted or rewritten
- The **Roster**, **KPI Card**, **PieChart Card**, **Total Weekly Income**, **Weekly Count**, **Daily Count**, and **Monthly Count** all operate on the **World Lens**-filtered **Mule** list; they never see **Mules** outside the **Selected World**
- Drag-to-reorder computes indices against the unfiltered **Mule** list so cross-**World** ordering survives a lens switch; visual ordering within the **Selected World** is preserved because every dragged id shares that world
- The **Add Card** click calls `addMule(world.id)` when a **Selected World** exists; with no **Selected World** it mounts the **World Missing Banner** and does not append a **Mule**
- The **World Missing Banner** unmounts the instant `world` transitions to non-null — no timer, no close affordance, no fade-out; unmount is the dismiss
- **World Weekly Crystal Cap** is a **Soft Cap** enforced at the **WorldIncome** layer — the user may select more than 180 **Crystal Slots'** worth of bosses, but the **World Cap Cut** drops the lowest-**Slot Value** slots until exactly 180 remain in the **World Slot Pool**. Distinct from **Weekly Crystal Cap**, a **Hard Cap** that blocks input on `MuleBossSlate.from`
- The **World Cap Cut** runs per **Selected World** in isolation — every **World** has its own independent **World Slot Pool** and its own 180-slot ceiling; a player with **Mules** across multiple **Worlds** has no cross-world budget
- The **Active-Flag Filter** runs upstream of the **World Cap Cut** — **Inactive Mules** never enter the **World Slot Pool**, so toggling a **Mule** inactive can free **Crystal Slots** for previously-dropped slots elsewhere in the **Roster**
- A **Daily Cadence** **Slate Key** contributes 7 **Crystal Slots** to the **World Slot Pool** but 0 keys to the per-**Mule** **Weekly Crystal Cap**; the per-**Mule** cap counts only **Weekly Cadence** keys, while the **World Weekly Crystal Cap** counts both cadences as **Crystal Slots**
- A **Monthly Cadence** **Slate Key** contributes 0 **Crystal Slots** and 0 meso to **WorldIncome**, mirroring its 0 contribution to **Total Crystal Value**; monthly selections are orthogonal to both cap pipelines
- The **KPI Card's** bignum reads `WorldIncome.totalContributedMeso`, not the sum of **Active Mules'** **Potential Meso** — the two diverge whenever the **World Cap Cut** drops at least one slot
- The **KPI Card's** WEEKLY and DAILY tiles read `WorldIncome.weeklySlotsContributed` and `WorldIncome.dailySlotsContributed` (post-cut counts); their sum is bounded by **World Weekly Crystal Cap** at all times
- The **Weekly Cap Rail's** numerator is `WorldIncome.slotsTotalContributed` (clamped to `≤ 180`); over-selection is invisible at the rail and surfaces only via **Cap Drop Badges**
- The **PieChart Card's** slices size on **Mule Contribution.contributedMeso**, not **Potential Meso** — a **Mule** whose **Crystal Slots** are entirely dropped by the **World Cap Cut** renders no slice
- A **Character Card's** headline meso reads **Potential Meso** (uncapped); when **Dropped Meso > 0**, the **Cap Drop Badge** appears alongside as the only per-**Mule** signal of the cut
- `Income.of()` is **World Cap Cut**-unaware — it returns per-**Income Source** **Potential Meso**; aggregate cap-aware surfaces (KPI bignum, rail, pie, WEEKLY/DAILY tiles) read **WorldIncome** instead. **Character Cards** keep using `Income.of()` for stable per-mule planning numbers
- The **Cap Tiebreak** is deterministic by construction: stable mule index in the **Roster**, then stable insertion order within `selectedBosses[]`. The **World Cap Cut** never flickers across unrelated toggles
- A **Boss Slate** is the validated representation of `Mule.selectedBosses`; constructing one via `MuleBossSlate.from(keys)` normalizes duplicates (highest-**Crystal Value** wins per `(bossId, cadence)`) and drops unresolvable or **Legacy Slate Key** entries silently
- The **Selection Invariant** is enforced by construction — callers cannot hold an invalid **Boss Slate**; the pre-refactor two-phase "toggle produces invalid intermediate, `useMules` re-validates" protocol is gone
- `MuleBossSlate.toggle(key)` returns a new **Boss Slate**: same key → deselect; different **Boss Difficulty** on the same `(bossId, cadence)` → **Tier Swap**; different **Boss Cadence** → both **Slate Keys** coexist
- `MuleBossSlate.view(search?)` projects a **Boss Slate** into `SlateFamily[]` with per-row `selected: boolean`; the **Boss Matrix** consumes this directly, replacing the old parallel `selectedKeys: string[]` prop
- **Weekly Count** is a lazy accessor on the **Boss Slate** — the count of **Slate Keys** whose **Boss Cadence** is `weekly`
- **Total Crystal Value** is a lazy accessor on the **Boss Slate**; `useIncome` (or its successor) multiplies it by **Party Size** and gates on **Active Flag** to produce **Potential Income**
- The **Boss Slate** does not own **Active Flag** or **Party Size** gating — those are caller concerns at the income pipeline
- `Income.of(mule, abbreviated)` delegates per-mule arithmetic to `MuleBossSlate.from(mule.selectedBosses).totalCrystalValue`; the income module owns aggregation, not crystal summation
- `Income.of(mules, abbreviated)` sums each **Income Source**'s contribution after applying the **Active-Flag Filter** — `active === false` drops, `true` and `undefined` pass
- `useIncome(source)` must be called inside an **Income Provider**; passing no argument returns an **Income** of `0` alongside `abbreviated` and `toggle`, useful for toggle-only consumers
- `useIncome` memoizes the returned **Income** — callers do not wrap in their own `useMemo`
- The **KPI Card**'s bignum pairs `useIncome(mules)` with `useAutoFullFormatOnZero(total.raw)`; the **Auto-Fullformat-On-Zero Rule** is extracted from the old inline `useEffect` and is the only caller of `useAutoFullFormatOnZero`
- **PieChart Card** slices use `Income.of(mule, false).raw` directly — chart math bypasses **Format Preference** because charts need numbers, not strings
- The **Mule Store** is the only module that reads or writes `Mule.selectedBosses` from storage; `useMules` consumes `store.load()` as its `useState` initializer and calls `store.save(mules)` inside a `useEffect` on every mutation
- `store.save(mules)` coalesces bursts over the **Storage Debounce**; `store.flush()` writes immediately regardless of pending debounce state
- `useMules` registers `store.flush` on `pagehide` and `beforeunload` and calls `store.flush()` in the effect cleanup so tab close never loses pending writes
- The **Storage Port** is the only outbound seam — tests construct `createMuleStore(fakePort)` and drive every **Load Mode** branch (v1 **Wipe**, v2 **Upgrade V2**, v3/v4 **As-Is Load**, corrupt JSON **Wipe**, unknown **Schema Version** **Wipe**) by reading fixture JSON through the fake
- The **Default Storage Port** engages the **Storage Fallback Ladder** on both read and write — `localStorage.getItem` throw falls through to `sessionStorage.getItem`; `localStorage.setItem` throw falls through to `sessionStorage.setItem`; both throwing is swallowed so React state remains the in-process source of truth
- **Mule Migration** is synchronous and pure — `muleMigrate(raw)` has no React, no `window`, no timers; it is the unit-testable core the **Mule Store** wraps with I/O and lifecycle
- The **Active Default** is applied inside **Mule Migration**, not in the React hook — any **Mule** missing an **Active Flag** on load becomes an **Active Mule**, preserving its **Potential Income** contribution under the new **Active-Flag Filter** semantics
- **Upgrade V2** resolves each v2 **Slate Key** through the **Boss Cadence** catalog to produce a v4 **Slate Key**; unresolvable entries drop silently per the same invariant that governs `MuleBossSlate.from`
- The public React API of `useMules` does not change across the **Mule Store** extraction — `{ mules, addMule, updateMule, deleteMule, deleteMules, reorderMules }` remains the contract for every consumer component
- A **Draft Field** `onChange` only mutates local `draft` state — no parent re-render until `onBlur` triggers **Commit**; this is what keeps the **PieChart Card** donut and **Roster** cards still during per-keystroke name/level editing
- **Commit** fires on three boundaries: `onBlur`, **Mule Switch**, and **Drawer Close**; all three run through the same `commit` callback so there is one code path for "push draft to parent state"
- **Draft Source Resync** runs inside `useDraftField`'s effect — if the external `source` changes while the user is editing, the draft rebaselines to the new source rather than silently shadowing it; the `equals` option lets callers opt out for structural values
- **Commit On Exit** unifies the pre-refactor `draftsRef` + `lastMuleIdRef` + `onUpdateRef` dance; initial mount does not fire a flush (first effect run seeds the tracking refs) so opening the **Drawer** never emits a spurious `onUpdate(id, {})`
- The **Drawer Close** path was a silent bug before the **Drawer Editor Hooks** refactor — the pre-refactor effect only flushed on **Mule Switch**, so Esc/backdrop close discarded any un-blurred edit; **Commit On Exit**'s unmount-flush closes this leak
- **Mule Identity Draft** exposes the level field as both `draft: string` (what the `<input>` renders) and `displayNumber: number` (what badges and summaries read), so the **Character Card**'s level badge never sees an in-flight non-numeric string
- **Boss Matrix View** is the only consumer of `MuleBossSlate.from(mule.selectedBosses).toggle(key)` inside the **Drawer** — the hook holds the **Slate Toggle** dispatch so the JSX only passes `toggleKey` down to the **Boss Matrix**
- **Preset Swap** emerges from **Conform** rather than existing as a separate reducer step: wiping every weekly key outside the clicked preset's entries inherently drops the prior **Active Preset**'s keys in the same atomic `onUpdate`. The historical "strip then apply" pair-step is gone
- Under **Same-Cadence Equality** the LOMIEN-supersedes-CRA tiebreaker is unreachable — a 14-entry selection can't also equal an 11-entry selection — so the `activePresets` set collapses to a single `activePill: PresetKey | null`
- **Delete Confirm** is the only **Drawer Editor Hook** that does not interact with **Mule** mutation directly — it is a confirmation guard in front of the injected `onDelete` callback; calling `confirm()` forwards to `onDelete(muleId)` and optionally fires `onAfterDelete` (typically `onClose`) so the **Drawer** closes after a delete
- The **Drawer** component becomes a pure wiring layer after the **Drawer Editor Hooks** refactor — zero `useState`, `useRef`, `useMemo`, or `useCallback` at the component level; every stateful concern lives inside one of the five hooks
- **Monthly Cadence** today applies only to Black Mage Hard and Extreme — every other **Boss Family** stays **Weekly Cadence** / **Daily Cadence**. The **Monthly Radio Mutex** is therefore scoped to one family; generalizing to additional families is premature until another family flips
- The **Monthly Radio Mutex** is not a new code branch — it falls out of the existing `(bossId, cadence)` **Selection Invariant**. Black Mage Hard and Extreme share the same `(bossId, 'monthly')` bucket, so toggling one while the other is selected naturally **Tier Swaps** and leaves **Monthly Count** at `1`
- A stale Black Mage `:hard:weekly` or `:extreme:weekly` **Slate Key** carried over from pre-monthly persistence fails to resolve (boss data says the tier is monthly now) and is silently dropped by `MuleBossSlate.from`. There is no migration — the silent-drop was an explicit grill decision to avoid write coupling for a pre-user cohort
- The **Crystal Tally** replaces the pre-refactor vertical two-row stack with a single horizontal three-cell layout; the old layout is gone (no orientation prop). Its three **Crystal Cells** share the tally's flex container evenly and are separated by two vertical **Gradient Hairlines**
- Under the **Cadence Filter**, a **Monthly Cadence** family is hidden by `Weekly` and `Daily` (none of its rows match the filter predicate) and visible only under `All` — no new filter code was required, just the data flip on Black Mage's tiers
- On touch, **Gesture Engagement** for both reorder drag and **Drag-to-Select** requires the **Long-Press Gate** to elapse (250ms, tolerance 5px); on mouse it is immediate at `pointerdown`
- **Tolerance Cancel** during the **Long-Press Gate** prevents **Gesture Engagement** and lets the browser's native scroll proceed — a user's vertical swipe over a **Character Card** scrolls the page as expected
- During an engaged **Drag-to-Select** on touch, pointer position inside the **Autoscroll Zone** triggers **Autoscroll Ramp** and **Continuous Paint** simultaneously — the page scrolls while new **Mules** entering the finger's coordinates adopt the **Brush**
- The **Drag Boundary** receives `touch-action: none` only while a touch-initiated **Drag-to-Select** is engaged, so the browser's native scroll does not fight **Autoscroll Ramp**; pre-engagement and post-release the default `touch-action` is restored
- The **Press-and-Hold Affordance** is scoped to touch input only — desktop mouse press does not scale the **Character Card**, preserving the existing click feel
- Keyboard reorder via **Keyboard Sensor** uses `Space` to pick up and drop, arrows to move, `Esc` to cancel — the a11y path for users without a pointer
- **Pre-Engagement** allows the browser's native scroll: the **Scroll Preventer** holds `preventDefault()` while `brushRef.current === null`, and **Tolerance Cancel** tears the **Long-Press Gate** down if the user drifts past 5px so the browser owns the gesture from there
- **Post-Engagement** blocks the browser's native scroll: the **Scroll Preventer** calls `preventDefault()` on every `touchmove`, which is the only mid-gesture escape hatch from the **Scroll Latch** that iOS Safari and Android Chrome honor
- Without the **Scroll Preventer**, reactive `touch-action: none` applied at **Gesture Engagement** cannot reverse the **Scroll Latch**'s `touchstart` commit, producing **Scroll Takeover** — the observable bug where **Start Card** flips selected and then un-flips as `pointercancel` fires and native scroll begins
- The **Scroll Preventer** is registered only when `e.pointerType === 'touch'` at `pointerdown`; mouse and pen paths do not attach it and are unaffected
- The **Scroll Preventer** is detached in `resetGesture` alongside the existing document-level `pointermove` / `pointerup` / `pointercancel` listeners, so every exit path (commit, cancel, mid-gesture unmount) tears down listener registration together
- A successful **Character Lookup** writes a **Real Avatar** **Avatar URL** onto the **Mule** through the same `onUpdate` channel a manual **Drawer** edit uses — there is no second persistence path
- The **Cloudflare Worker** sits between the SPA and Nexon's upstream — the SPA never calls Nexon's ranking API directly, but it **does** load the **Real Avatar** PNG straight from Nexon's CDN (the **Worker** never proxies image bytes)
- The **Lookup Cache** is consulted before the **Nexon Adapter** on every lookup; cache misses drop through the adapter and write a fresh entry on the way back
- The **World ID Map** is consulted twice per cache miss — once to choose the **Region** + **Reboot Index** + **Upstream Base URL** for the upstream call, once to filter the response `ranks[]` by Nexon's numeric **worldID** so a same-named character on a different **World** is rejected as 404
- A **Mule's** **World Id** drives both the **World Lens** (which **Mules** are visible in the **Roster**) and the **Character Lookup** scoping (which **Region** + **Reboot Index** + which numeric **worldID** to expect) — one immutable field, two distinct roles
- **Real Avatar** URLs persist on the **Mule** via the existing **Storage Debounce** + **Storage Port**; no schema-version field tracks "last looked up"

> **Domain expert:** "That's the single-click fallback. Unresolvable or **Legacy Slate Key** entries drop silently. What comes out is a valid **Boss Slate** — invalid states are unrepresentable downstream."

> **Domain expert:** "Right. `slate.toggle(normalKey)` returns a new **Boss Slate** with the Hard **Slate Key** replaced by the Normal one — same `(bossId, cadence)`, so the **Selection Invariant** forces the swap. > **Domain expert:** "The **Boss Slate** exposes **Total Crystal Value** as a lazy accessor — daily **Slate Keys** folded × 7, weekly × 1. That's the raw crystal-value basis. The income pipeline multiplies by **Party Size** and gates on **Active Flag** to produce **Potential Income**, then sums **Active Mules** for **Total Weekly Income**. Gating lives at the caller, not on the slate."
> **Dev:** "So when I call `useIncome(mules)` from the **KPI Card**, what exactly comes back?"
> **Domain expert:** "An **Income** — `raw` is the **Total Weekly Income** as a number after the **Active-Flag Filter**, `formatted` is the same number rendered per **Format Preference**. Plus `abbreviated` (the current preference) and `toggle` to flip it. One object for display, math, and UX controls."

> **Domain expert:** "In **useAutoFullFormatOnZero**, a named hook. The **KPI Card** calls `useAutoFullFormatOnZero(total.raw)` right after `useIncome(mules)`. That's the whole **Auto-Fullformat-On-Zero Rule** — one hook, one effect, one test."
> **Dev:** "If I use **Income** from a chart — say a donut slice — do I need a provider?"
> **Domain expert:** "Not for the math. Call `Income.of(mule, false).raw` directly; it's a pure static factory. The **Income Provider** only exists for React-side **Format Preference** plumbing. Charts skip formatting entirely."

> **Domain expert:** "No — it moved. **Mule Migration** owns the whole **Schema Lineage**. `muleMigrate(raw)` is pure: it reads the **Persisted Root**, picks a **Load Mode** off the **Schema Version**, runs **Upgrade V2** or **As-Is Load** or **Wipe**, applies the **Active Default**, and returns `Mule[]`. `useMules` never sees a **Load Mode** enum."
> **Dev:** "So if the v2 upgrade can't resolve a **Slate Key's** **Boss Cadence**, it crashes?"
> **Domain expert:** "It drops the key silently — same contract as `MuleBossSlate.from`. Fail-safe everywhere on the load path. The worst that happens is a **Mule** comes back with fewer selected bosses than it had on disk; we never surface a parse error to the user."
> **Dev:** "What's `createMuleStore(port?)` for? I thought persistence just talked to localStorage directly."
> **Domain expert:** "The **Storage Port** is how the **Mule Store** decouples from the browser. The **Default Storage Port** is `localStorage` plus `sessionStorage` as the fallback — that's the **Storage Fallback Ladder**. In tests you pass an in-memory fake port and you can exercise every **Load Mode** without JSDOM. Future cloud-sync would be another **Storage Port** implementation."
> **Dev:** "So `store.save(mules)` writes every render?"
> **Domain expert:** "No — `save` coalesces through the **Storage Debounce**, 200ms. Ten rapid setState calls produce one `port.write`. If the user closes the tab mid-debounce we still get the last state because the `pagehide` listener calls `store.flush()`, which drains any pending write immediately."
> **Dev:** "And `useMules`'s shape changes for consumers?"
> **Domain expert:** "Zero change. Every consumer — `App`, `KpiCard`, `Roster`, `Drawer` — still destructures `{ mules, addMule, updateMule, deleteMule, deleteMules, reorderMules }`. The refactor is internal: the hook became a CRUD facade over the **Mule Store**. All the **Schema Version** literals, debounce refs, and storage keys moved out of the hook file."
> **Dev:** "If the user types a new mule name and hits Esc, does the name save?"
> **Domain expert:** "Yes, now. That's **Commit On Exit**. Before this refactor, Esc and backdrop-click dropped any unblurred **Draft Field** silently — the effect only flushed on **Mule Switch**. The new hook flushes on unmount too, so **Drawer Close** commits drafts the same way `onBlur` would. The name lands before the **Drawer** goes away."
> **Dev:** "And per-keystroke in the name field — does that re-render the **PieChart Card** donut?"
> **Domain expert:** "No. **Draft Field**'s `onChange` only touches local `draft` state. The parent `mules` array doesn't move until **Commit**, so the donut and the **Roster** stay put while the user is typing. That's the whole reason we kept the draft/commit split in the hooks refactor."
> **Dev:** "When I click **CTENE** while **CRA** is already an **Active Preset**, do both stay on?"
> **Domain expert:** "No — **Preset Swap**. The **Boss Matrix View** hook strips the **CRA** keys before applying the **CTENE** keys, in one `onUpdate` call. The user sees a clean swap, not a union. If **CTENE** were a strict superset of **CRA**, the final **Active Presets** would show only **CTENE** — that's the LOMIEN-supersedes-CRA case."
> **Dev:** "And **Delete Confirm** — if the user hits delete, cancels, then switches to another **Mule**, does the confirm state persist?"
> **Domain expert:** "It auto-resets on **Mule Switch**. **Delete Confirm** watches `muleId`; any change clears `confirming` so the next **Mule** starts clean. The guard is per-open-mule, never sticky across switches."
> **Dev:** "The **World Select** in the header — when I pick Kronos, does the **Roster** filter to Kronos **Mules**?"
> **Domain expert:** "Yes — that's the **World Lens**. Every **Mule** carries an immutable **Mule World Id** set at creation, and the lens predicate `mule.worldId === selectedWorld.id` drives the filter. The **Roster**, the **KPI Card**, the **PieChart Card**, and every count all see only the **Selected World**'s **Mules**. Switch from Kronos to Hyperion and the whole view swaps — including the **Weekly Cap Rail's** **Crystal Sale Tally**."
> **Dev:** "Why is **World Provider** outside **Income Provider**?"
> **Domain expert:** "Because the **World Lens** feeds **Crystal Values** that depend on **World Group** (**Heroic** vs **Interactive** have different crystal economies). Income needs to read `useWorld()`, so **World Provider** has to sit above **Income Provider** in the tree."
> **Dev:** "And if I click **Add Card** on a fresh install with no **World** picked?"
> **Domain expert:** "The **World Missing Banner** slides in above the **Roster Header**. Blue info-variant styling, same `bulk-slide` keyframe as the **Bulk Action Bar**, text reads 'Please select a world first' No **Mule** is appended. Pick any **World** from the **World Select** and the banner unmounts — unmount is the dismiss, no close button, no timer."
> **Dev:** "Can I move a **Mule** from Kronos to Hyperion later if I picked wrong?"
> **Domain expert:** "No — **Mule World Id** is immutable. The game's own world-transfer is a rare out-of-game event, so the tracker mirrors it: wrong world means delete and recreate. The **Drawer** has no world editor."
> **Dev:** "The **KPI Card** shows WEEKLY CAP as `X/180` but the **Crystal Tally** shows `X/14` — same word, different cap?"
> **Domain expert:** "Right — different scopes. **Weekly Crystal Cap** (`14`) is per-**Mule** and lives in the **Crystal Tally** inside the **Drawer**; it's enforced as an invariant on `MuleBossSlate.from`. **World Weekly Crystal Cap** (`180`) is per-**Selected World** and lives in the **KPI Card**, summing **Weekly Count** + **Daily Count** across every **Active Mule** in the current **World Lens**; it remains a display reference, not an enforced cap."
> **Dev:** "And if someone sets `localStorage.world = 'garbage'` in devtools?"
> **Domain expert:** "**World Provider** validates on init against the **World Ids** set. Unknown value → **Selected World** falls back to `null`, the **World Select** shows its `Select world` placeholder, and the app renders normally."
> **Dev:** "Black Mage Hard is 4.5B and Extreme is 18B — if I have Hard selected and then click Extreme, do I get both?"
> **Domain expert:** "No — you get a **Tier Swap** under the **Monthly Radio Mutex**. Both sit in the same `(black-mage-id, 'monthly')` bucket, so the **Selection Invariant** forces one-wins. **Monthly Count** stays at `1`, and the **Crystal Tally's** Monthly cell keeps reading `1/1`. If you click Extreme a second time it deselects and you drop to `0/1`."
> **Dev:** "So the **Monthly Crystal Cap** of `1` is enforced — and the **Weekly Crystal Cap** of `14` is too?"
> **Domain expert:** "Right — both are enforced on the **Boss Slate**, neither by the **Crystal Tally** component itself. Monthly: `MuleBossSlate.from` picks the highest-**Crystal Value** winner per `(bossId, cadence)` bucket via the **Selection Invariant**, which collapses Black Mage Hard + Extreme to one. Weekly: after that dedupe, `MuleBossSlate.from` runs a second trim that drops the lowest-`crystalValue` weeklies until at most 14 remain (ties broken by insertion order). By the time the count reaches the tally it's already clamped to ≤14."
> **Dev:** "What happens to a saved `:hard:weekly` on Black Mage after the cadence flip?"
> **Domain expert:** "Silent drop. The key no longer resolves — `parseKey` sees Hard Black Mage is monthly in the catalog, which disagrees with the `:weekly` suffix. `MuleBossSlate.from` discards it. The **Mule** reloads with one less selected boss. That was an explicit grill decision — no migration, accept the loss for the pre-user cohort."
> **Dev:** "And the **Crystal Tally** — where did it go? I remember it being stacked vertically."
> **Domain expert:** "Replaced. Today's layout is three **Crystal Cells** side-by-side in **Crystal Tally Order** (Weekly, Daily, Monthly), each with icon on top, count in the middle, label at the bottom, separated by two vertical **Gradient Hairlines**. No orientation prop — there's a single code path. Narrow-viewport behaviour is a future concern."
> **Dev:** "If I select Black Mage Extreme, why does the weekly-mesos pill drop instead of going up by 18B?"
> **Domain expert:** "That's the **Monthly Income Regression**. **Total Crystal Value** weights monthly × 0 on purpose — there's no honest weekly-basis multiplier for a once-a-month kill, and the grill deferred a dedicated monthly income readout to a follow-up. The pill stays labelled 'Weekly' and tells the truth about **Weekly Cadence** income only. The Monthly cell in the **Crystal Tally** is where monthly selection shows up today."
> **Dev:** "Under the `Weekly` **Cadence Filter**, does Black Mage still show up in the **Boss Matrix**?"
> **Domain expert:** "No — Black Mage's rows are all `monthly`, so the filter predicate `some(row.cadence === 'weekly')` returns false and the family is hidden. Same under `Daily`. Only `All` shows Black Mage now. And there's no `Monthly` chip in the **Matrix Toolbar** — by design, since it'd filter to a single family."
> **Dev:** "On my phone, why can I scroll past a **Character Card** without accidentally dragging it, but a slow hold starts a drag?"
> **Domain expert:** "That's the **Long-Press Gate** — 250ms timer after `pointerdown`, with 5px **Tolerance Cancel**. Move your finger more than 5px before the timer fires and we back off so the browser can scroll. Hold still for 250ms and **Gesture Engagement** fires — you'll see the **Press-and-Hold Affordance** scale-up first, then the card snaps into the **DragOverlay**."
> **Dev:** "And in **Bulk Delete Mode** when I drag my finger to the bottom of the screen?"
> **Domain expert:** "The **Autoscroll Zone** kicks in — 100px from the viewport edge. Inside that zone the **Autoscroll Ramp** scrolls the page at 0 → 14 px/frame, and **Continuous Paint** re-samples `elementFromPoint` every frame so **Mules** coming under your stationary finger get added to the **Paint Range** automatically."
> **Dev:** "Does the same **Long-Press Gate** apply on desktop?"
> **Domain expert:** "No — **Mouse Sensor** still uses `distance: 0`, so mouse drag engages instantly. Only **Touch Sensor** has the 250ms delay. Mouse users don't need to re-learn anything."
> **Dev:** "On my iPhone, the **Start Card** selects at 250ms but the moment I start dragging it deselects and the page scrolls — what happened?"
> **Domain expert:** "That's **Scroll Takeover**, driven by the **Scroll Latch**. Safari committed to scroll back at `touchstart` because `touch-action` was the default `auto` at that instant. The reactive `touch-action: none` we flip at **Gesture Engagement** arrives too late — iOS has already decided. The browser fires `pointercancel` as it begins scrolling, and **Gesture Cancellation** reverts the **Original Snapshot**, un-flipping the **Start Card**."
> **Dev:** "So `touch-action: none` is useless?"
> **Domain expert:** "It's defense-in-depth for non-latching browsers, but it can't solve the **Scroll Latch** on its own. The fix is the **Scroll Preventer** — a non-passive native `touchmove` listener on `document`, registered at `pointerdown`, that calls `preventDefault()` only in **Post-Engagement**. `preventDefault` on touchmove is the one lever iOS re-evaluates mid-gesture, so the moment **Gesture Engagement** fires the scroll is cancelled per-event."
> **Dev:** "Why gate on **Post-Engagement**? Why not block scroll the whole time?"
> **Domain expert:** "**Pre-Engagement** is how the user scrolls a long **Roster** in **Bulk Delete Mode** — tap-and-drift lets **Tolerance Cancel** abandon the gesture and the browser scrolls normally. If the **Scroll Preventer** called `preventDefault` before **Gesture Engagement**, you'd lose the ability to scroll through the **Roster** to find cards to paint. Gating on `brushRef.current !== null` preserves that pre-engage scroll."
> **Dev:** "I tried to add a 15th **Weekly Cadence** boss to a **Mule** and the **Crystal Tally** still reads `14/14`. Bug?"
> **Domain expert:** "That's the **Weekly Crystal Cap** invariant. `MuleBossSlate.from` trims any over-cap weekly **Slate Keys** at slate construction — it keeps the 14 highest-`crystalValue` weeklies (ties broken in favour of the earlier-inserted key) and drops the rest. **Daily Cadence** selections still contribute `crystalValue × 7` and stay uncapped at the per-mule layer, and **Monthly Cadence** still contributes `0` regardless. **Total Crystal Value** is now a plain sum of the slate's weeklies plus the daily total — the old **Top-14 Weekly Cut** sort-and-slice is retired because the cap is held upstream."
> **Dev:** "What about the world-level cap? My roster has 200 weekly+daily **Crystal Slots** in Kronos but the **KPI Card** bignum looks weirdly low and my last **Mule** has a `−18M to cap` badge."
> **Domain expert:** "That's the **World Cap Cut**. **World Weekly Crystal Cap** (`180`) is a **Soft Cap** — selecting past it is allowed, but `WorldIncome.of(mulesInWorld)` builds a **World Slot Pool** from every **Active Mule's** **Crystal Slots**, sorts by **Slot Value** descending, takes the top 180, and treats the rest as zero. The bignum reads `WorldIncome.totalContributedMeso`, so the dropped slots' meso is gone from the headline. The **Cap Drop Badge** on that **Mule** is its `droppedMeso` from **Mule Contribution** — it tells you which **Mule** ate the loss without changing its **Potential Meso** on the card itself."
> **Dev:** "Why does the rail just say `180/180` instead of `200/180` like it used to?"
> **Domain expert:** "Because the **Crystal Sale Tally** is now `WorldIncome.slotsTotalContributed`, not raw selection count. After the **World Cap Cut** you can never have more than 180 contributing **Crystal Slots** by definition, so the rail clamps cleanly. The pre-cut behaviour (`185 / 180 · 100%`) was the right answer when the cap was advisory — you needed the count to leak the overshoot. Now the **World Cap Cut** drops the lowest-**Slot Value** slots silently and the rail just shows the bucket as full."
> **Dev:** "If two **Crystal Slots** have the exact same **Slot Value** at the boundary, which one drops?"
> **Domain expert:** "**Cap Tiebreak**. The slot whose owning **Mule** has the higher **Roster** index drops first — so freshly-added **Mules** lose ties. Inside one **Mule**, the slot whose **Slate Key** appears later in `selectedBosses[]` drops first. Both axes are stable, so unrelated toggles never flip which **Mule** holds the badge."
> **Dev:** "And a **Daily Cadence** Hilla — does that count as 1 slot or 7?"
> **Domain expert:** "Seven, each at base `crystalValue` (party size ignored, matching `MuleBossSlate.totalCrystalValue`'s daily rule). So the cap can drop a daily _partially_ — 5 of Hilla's 7 days might fall out while 2 stay in. The mule still has Hilla 'selected' on the **Boss Matrix**; the loss surfaces as **Dropped Slots: 5** in the **Mule Contribution** and `−20M to cap` on the **Cap Drop Badge**. The matrix cell itself doesn't change colour — the **Boss Matrix** is a selection surface, not a contribution surface."
> **Dev:** "And the **PieChart Card** — if a **Mule** has every slot dropped, do I still see a slice?"
> **Domain expert:** "No. Slices size on **Mule Contribution.contributedMeso**, not **Potential Meso**. A 100%-dropped **Mule** has `contributedMeso === 0` and renders no slice. You'd find them on the **Roster** with a full-headline **Potential Meso** and a **Cap Drop Badge** equal to that potential. The pie and the **KPI** agree on the same post-cut total; the **Character Cards** keep their stable potential numbers for planning."
> **Dev:** "Why does the same **Mule** I just copied over from Kronos to Scania now show ~1/5 the weekly mesos?"
> **Domain expert:** "That's **World Pricing** working. Kronos is a **Heroic** world; Scania is **Interactive**. Every **Boss Difficulty's** **Crystal Value** stores both a **Heroic Crystal Value** and an **Interactive Crystal Value** — and `Income.of` calls **Resolve World Group** on the **Mule's** **World Id** to pick which one to sum. The **Crystal Value Ratio** is ~0.2, so Interactive income is roughly a fifth. Same selections, different economy."
> **Dev:** "Is it always exactly 5×?"
> **Domain expert:** "No — **Extreme Kaling** is the outlier. Its **Heroic Crystal Value** is `6 026 000 000` but the **Interactive Crystal Value** is `1 205 250 000`, not the 5×-exact `1 205 200 000`. A 250 000-meso deviation. That's why we store both values explicitly and don't assert the ratio in tests — future bosses may deviate more."
> **Dev:** "And a **Mule** I created before the **World Select** feature landed — no **World Id** set yet?"
> **Domain expert:** "**Fallback World Group** kicks in — **Heroic**. Its income is identical to what it was pre-**World Pricing**. The user only sees a change when they assign an Interactive **World** to that **Mule** explicitly."
> **Dev:** "When I click **Lookup Button** on a Kronos **Mule** named 'Pinkbean', what actually goes out the wire?"
> **Domain expert:** "The browser sends `GET /api/character/Pinkbean?worldId=heroic-kronos` — same-origin via the **Same-Origin Proxy**. Nginx forwards it to the **Cloudflare Worker**. The **Worker** checks the **Lookup Cache** keyed on `('pinkbean', 'heroic-kronos')` and on miss runs the **World ID Map** to get `{ region: 'na', rebootIndex: 1, worldID: 45 }`. The **Nexon Adapter** picks `NEXON_RANKING_BASE_URLS['na']` and queries the **Weekly Ranking** at `/v2/na?type=overall&id=weekly&reboot_index=1&character_name=Pinkbean`. We filter the response `ranks[]` for the entry whose numeric **worldID** equals `45` — that disambiguates a same-named character on Hyperion. The matching entry's `characterImgURL` becomes the **Real Avatar** **Avatar URL** the SPA persists on the **Mule**."
> **Dev:** "What if Pinkbean exists on Kronos but hasn't logged in this week?"
> **Domain expert:** "The **Weekly Ranking** doesn't index them — `ranks[]` comes back empty, the **Worker** returns 404, and the not-found toast explains the **Ranking Window**: 'Character not found in rankings — must be logged in within the last week.' We cache that 404 for an hour so a typo isn't a re-hammer on Nexon. The **Mule's** existing fields are untouched — failed lookups never mutate."
> **Dev:** "And the avatar PNG itself — does it stream through the **Worker**?"
> **Domain expert:** "No. **Avatar URL** points straight at `msavatar1.nexon.net`. The **Worker** only handles JSON. The browser fetches the PNG cross-origin from Nexon's CDN, the browser's HTTP cache handles repeat loads, and the **Worker** never sees image bytes. If we'd proxied bytes we'd be paying Cloudflare egress for every avatar render — totally unnecessary."
> **Dev:** "What changes when I look up a **Solis** character versus a **Kronos** one?"
> **Domain expert:** "**Region**. The **World ID Map** says **Solis** is `(region: 'eu', rebootIndex: 1, worldID: 46)`, so the **Nexon Adapter** picks `NEXON_RANKING_BASE_URLS['eu']` and queries `/v2/eu?type=overall&id=weekly&reboot_index=1&...`. Kronos goes through `/v2/na` instead. Same flow, different **Upstream Base URL** — that's the only thing **Region** controls. The post-fetch filter still uses the numeric **worldID** to pin the right character row."
> **Dev:** "Could the same numeric **worldID** mean different worlds across regions?"
> **Domain expert:** "Yes — every **Region** has its own numeric namespace inside each **Reboot Index** bucket. Today's six **Worlds** happen to be tuple-unique without **Region** in the key, but that's coincidence; uniqueness is structurally guaranteed only inside `(region, rebootIndex)`. The map carries **Region** explicitly so a future seventh **World** can't silently collide."
> **Dev:** "Same name on **Kronos** and **Hyperion** — could the **Worker** ever return Hyperion's by mistake?"
> **Domain expert:** "No. Even though Nexon's response can include both, we filter `ranks[]` by the expected numeric **worldID** from the **World ID Map** before reshaping. A miss on the expected **worldID** is a 404, not a fallback. **Real Avatar** is always for the requested **World Id** or it's nothing."
> **Dev:** "And if Nexon ever changes their JSON shape?"
> **Domain expert:** "The **Nexon Adapter** is the only file that touches the upstream contract — fixture tests there will fail loudly the moment the response drifts. Everything downstream — the **Worker Endpoint**, **Lookup Result**, **Lookup Button** — speaks our own normalized contract and is insulated from upstream churn."

## Flagged ambiguities

- "Income" was used inconsistently — sometimes meaning a mule's individual total, sometimes the global sum. Canonical terms: **Potential Income** (per mule) and **Total Weekly Income** (all mules). **Resolved in code:** `calculateMuleIncome` → `calculatePotentialIncome`, `totalIncome` → `totalWeeklyIncome`, local `income` → `potentialIncome`.
- "Boss" was overloaded to mean both the encounter and the crystal it drops. Canonical separation: **Boss** = the encounter, **Crystal** = the sellable item, **Crystal Value** = the mesos received. **Resolved in code:** `mesoValue` → `crystalValue` on the `Boss` type and all boss data entries.
- "Card" is repeatedly overloaded — **Character Card**, **KPI Card**, **PieChart Card**, **Add Card** all coexist. Always use the full compound name; never write "the card" unqualified when more than one is in context.
- "Mule Card" (from the **Handoff** files) refers to the same concept as **Character Card**. Canonical term: **Character Card**. The component file is named `MuleCharacterCard.tsx` for historical reasons; the class name and type stay but prose and PR descriptions should say **Character Card**.
- "Dark mode" / "light mode" were used interchangeably with the **Theme** names. Canonical terms: **Dark-Amber** and **Cozy-Pastel**. "Dark mode" is acceptable shorthand when contrast with "light mode" is the point, but never imply a third option — only these two themes exist.
- "Accent" in CSS has multiple variables (`--accent`, `--accent-raw`, `--accent-soft`, `--accent-glow`, `--accent-primary`, `--accent-secondary`, `--accent-numeric`). Prose canonical: **Accent** (the raw source color), **Accent Soft** (hover tint), **Accent Glow** (shadow halo). The shadcn-compatible `--accent` / `--accent-primary` / `--accent-numeric` are all aliases of the raw accent; prefer `--accent-raw` in new code.
- "Layout" was briefly polysemous — the **Handoff** ships four layouts (Classic, Bento, Leaderboard, Sidebar) but this app supports only **Classic Layout**. Do not refer to the others except as historical reference.
- "Entry" and "slot" were used interchangeably. Canonical term: **Entry Slot**. Not yet in code (future enhancement for daily/weekly distinction).
- "Preset" is now ambiguous — distinguish **Mule Preset** (template for fast-creating mules, not yet in code) from **Boss Preset** (the four pills in the **Matrix Toolbar**: **CRA**, **LOMIEN**, **CTENE**, **Custom Preset**). Never write "preset" unqualified. Within **Boss Preset**, further distinguish **Canonical Preset** (CRA/LOMIEN/CTENE — click-actionable) from **Custom Preset** (inert indicator).
- "**Active Preset**" was previously subset-match semantics — a pill was active if every preset key appeared in the selection, tolerating extras. The canonical rule is now **Same-Cadence Equality**: strict exact match of weekly keys against **Preset Entries**' **Accepted Tiers**, with dailies orthogonal. Under the new rule adding a non-preset weekly to an **Active Preset** collapses it to **Custom Preset**; under the old rule the pill stayed lit. If you encounter code or prose that treats **Active Preset** as subset-match, it is pre-slice and needs updating.
- "Apply preset" / "toggle preset" / "select preset" were used loosely for the pill click. Canonical verb: **Conform**. The pill click no longer has a "deselect" gesture — the old Q3-style "re-click active pill to strip its keys" semantics are explicitly out of scope; the only exit paths are **Preset Swap** (to another **Canonical Preset**), **Matrix Reset**, or manual cell deselection.
- "Reset" was already flagged — now **Matrix Reset** joins **Weekly Reset**, **Reset Anchor**, and **Reset Countdown** in the set of distinct concepts. Always qualify: **Matrix Reset** clears selections, **Weekly Reset** is the game event, **Reset Countdown** is the header widget.
- "Filter" in prose could mean the **Cadence Filter** control or the `filterByCadence` / `filterBySearch` pipeline functions. Prefer **Cadence Filter** when referring to the UI; use function names when referring to code.
- "Hardest" / "top tier" / "max difficulty" — canonical term **Hardest Tier**, defined by `max(crystalValue)` on a boss's difficulty entries.
- "Drawer", "side drawer", "detail panel", and "modal" were all used to describe the right-side editing panel. Canonical term: **Drawer**.
- "Current day" and "today" were considered for a day-of-week label in the header but dropped from scope; avoid reintroducing either term in **Reset Countdown** copy. The **Reset Anchor** is always expressed as a duration, never as a day name.
- "Reset" alone is ambiguous — could mean **Weekly Reset** (the event), **Reset Anchor** (the instant), or **Reset Countdown** (the widget). In code and prose, pick the specific term.
- "Active" has been redefined. **Previous definition:** "a **Mule** with at least one **Boss Difficulty** selected" (income-derived). **New definition:** "a **Mule** whose **Active Flag** is `true`" (user-declared). The **Active Flag** is the explicit, persisted source of truth; code that previously tested `selectedBosses.length > 0` to infer activeness should be updated to test `mule.active`. Keep in mind **Active Mule** and **Inactive Mule** no longer correlate with having bosses — a brand-new **Active Mule** may have zero bosses and still count as active.
- New mules default to **Active Mule** (**Active Flag** `true`) on creation so a freshly-added **Mule** immediately counts toward **Total Weekly Income** as the user configures bosses. Existing persisted mules also migrate to **Active Mule** (**Active Flag** `true`) on schema upgrade to preserve their contribution.
- "Delete" is now polysemous — distinguish **Single Delete** (per-card hover popover or **Drawer** footer; operates on one **Mule**) from **Bulk Delete** (the **Bulk Confirm** action that removes every **Deletion-Marked Mule** in one pass). Never write "delete" unqualified when both paths are in scope.
- "Selected" is now polysemous — **Deletion-Marked Mule** is a **Mule** toggled on inside **Bulk Delete Mode** for removal; the "currently selected mule" in the **Drawer** is instead referred to as the "open **Mule**" or the "**Mule** being edited". Prefer **Deletion-Marked Mule** over "selected mule" in any **Bulk Delete Mode** context.
- "Paint" and "brush" in the **Drag-to-Select** code could suggest free-form painting semantics (i.e. toggle every cell the pointer enters). They don't — the actual behaviour is range-based and reversible. Canonical prose term: **Drag-to-Select**. Reserve "**Brush**" strictly for the polarity (`add`/`remove`) fixed at the start of a gesture, never for the gesture itself.
- "Roster Header" is the canonical name for the row above the **Roster** that contains title + density + bulk controls. It has two visual states — the default header and the **Bulk Action Bar** — but it is conceptually one slot.
- "Boss selection" / "selected bosses" / "boss keys" were used loosely across hooks, components, and tests. Canonical terms: **Boss Slate** for the whole validated collection, **Slate Key** for one entry. The persisted field `Mule.selectedBosses: string[]` is the raw storage shape; a **Boss Slate** is its validated, immutable, behaviour-bearing representation. Prefer **Boss Slate** in prose and PR descriptions; keep `selectedBosses` only when referring to the storage field itself.
- "World Id" is now polysemous across the **Character Lookup** boundary. **Our** **World Id** is the slug like `heroic-kronos` — the canonical form everywhere in the SPA and the **Worker Endpoint**. **Nexon's** **worldID** is a numeric integer like `19` returned in their ranking response. The two live on opposite sides of the **World ID Map**; never use them interchangeably in prose. Prefer **World Id** for ours and explicitly say "numeric **worldID**" or "Nexon's **worldID**" when referring to theirs.
- "Avatar" was previously unambiguous — the static `blank-character.png` placeholder. After **Character Lookup** ships, the **Character Avatar** rendering surface displays either a **Real Avatar** (when **Avatar URL** is present on the **Mule**) or the placeholder fallback. "Avatar" alone is now imprecise; prefer **Real Avatar** vs placeholder when the distinction matters, or **Character Avatar** when referring to the rendering surface itself regardless of which underlying image is shown.
- "Lookup" alone is overloaded across multiple subsystems: **Character Lookup** (the Nexon-fetch flow), `findWorld` / `Resolve World Group` (in-memory **World Id** resolution), and the **World ID Map** (slug ↔ numeric translation). Prefer **Character Lookup** for the Nexon flow and the function name for in-memory resolution; never write "lookup" unqualified when **Character Lookup** is in scope.
- "Reboot" appears only at the **Nexon Adapter** boundary as **Reboot Index**, an artifact of Nexon's historical naming for **Heroic** worlds. Outside that boundary, the **World Group** terms are **Heroic** and **Interactive**. Do not import "Reboot" as a synonym for "Heroic" anywhere else in code or prose; the **Reboot Index** is purely an upstream-facing translation.
- "Worker" alone is ambiguous between **Cloudflare Worker** (our proxy service) and any future Web Worker / service worker. Prefer **Cloudflare Worker** when the proxy is meant; reserve "the Worker" unqualified only when a single **Cloudflare Worker** is unambiguously in scope.
- "Cache" is now polysemous — the **Lookup Cache** (Cloudflare Cache API, ~6h success / ~1h not-found) lives inside the **Cloudflare Worker** and is invisible to the SPA; the SPA's only persistence is the existing **Storage Port** + **Storage Debounce**. Don't conflate the two; a stale **Real Avatar** URL on a **Mule** is a **Storage Port** concern, not a **Lookup Cache** concern.
- The **Weekly Ranking** constraint has user-facing prose implications. The not-found toast must explicitly mention the **Ranking Window** ("must be logged in within the last week") because users with brand-new or stale **Mules** otherwise assume the feature is broken. Generic "not found" copy is wrong; the constraint is part of the contract.
